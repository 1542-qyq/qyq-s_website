## 1. 项目背景

### 1.1 市场背景

- 中国陶瓷行业年天然气消耗超 **200亿m³**，窑炉能耗占生产成本 **30%-40%**
- 国家"双碳"政策驱动，陶瓷行业面临 **碳排放配额收紧** 与 **环保达标** 双重压力
- 传统窑炉燃烧控制依赖人工经验，存在 **调控粗放、响应滞后、燃烧效率低** 等痛点
- 工业物联网（IIoT）与AI预测技术在窑炉节能领域应用尚处早期，**垂直数字化解决方案市场空白**
- 单条窑炉年运营成本 **400万-800万元**，节能降耗 **10%-15%** 即可带来显著经济效益

### 1.2 项目目标

| 目标类型 | 具体目标              | 衡量指标                         |
| ---- | ----------------- | ---------------------------- |
| 商业目标 | 单窑炉天然气节约率 ≥ 10%   | （改造前日均消耗 - 改造后日均消耗）/ 改造前日均消耗 |
| 商业目标 | 燃烧效率提升至 ≥ 80%     | 实际输出功率 / 理论燃料热值              |
| 商业目标 | 项目签约客户 ≥ 20 家陶瓷厂  | 签约并部署系统的陶瓷企业数量               |
| 产品目标 | 烧嘴运行状态预测准确率 ≥ 92% | 预测值 vs 真实值的R²系数              |
| 产品目标 | 异常预警响应时间 ≤ 30秒    | 异常发生到系统发出预警的时间               |
| 产品目标 | 窑炉温度波动控制 ±5℃以内    | 目标温度与实际温度偏差的均方根              |

### 1.4 业务扩展背景

**硬件销售（智能节能烧嘴）**：

- 客单价适中（¥1,400/支），但需持续复购与维护
- 硬件是基础入口，建立客户信任与数据触点

**数字化服务（数据监测+预测+优化）**：

- 决策重、技术门槛高、信任门槛高，但客户生命周期价值（LTV）大
- 订阅制或按效果收费模式，形成持续收入

**协同效应**：

- 安装烧嘴 → 接入数字化平台 → 持续数据服务 → 工艺优化建议 → 复购/升级
- 形成"硬件+数据+服务"的终身价值闭环

**O2O模式**：

- 线上（小程序/PC端）实时监测、数据分析、预警推送
- 线下（技术团队）安装调试、定期巡检、故障处理

### 3.3 用户使用场景

**场景1：生产管理人员日常监控**

1. 打开微信小程序，查看今日能耗概览（天然气消耗、燃烧效率、温度曲线）
2. 发现3号窑炉燃烧效率下降至75%，点击查看详情
3. 查看BiLG-BLS模型预测：未来2小时效率将持续下降，建议调整空燃比
4. 点击"一键优化"，系统自动下发PID调节指令
5. 10分钟后查看优化效果，效率回升至80%
6. 收到系统推送的日报：今日节约天然气1,200m³，节约成本¥4,800

**场景2：高层决策者评估投资回报**

1. 登录PC端管理后台
2. 查看"节能效益看板"：本月累计节约天然气36,000m³，节约成本¥144,000
3. 查看ROI计算：系统投入¥150,000，预计10个月回本
4. 导出月度节能报告，用于内部汇报和政府补贴申请
5. 查看同行业对比：本厂燃烧效率排名区域前20%

**场景3：技术工程师远程诊断**

1. 收到系统告警：XX陶瓷厂5号烧嘴温度异常波动
2. 登录后台，查看该烧嘴近24小时运行数据
3. 分析传感器数据：燃气流量传感器读数异常，疑似故障
4. 远程调取边缘计算日志，确认传感器漂移
5. 联系客户安排更换传感器，指导临时手动调控参数
6. 更新设备维护记录，标记该批次传感器需关注

**场景4：一线操作工接收预警**

1. 微信小程序收到推送："2号窑炉温度异常升高，当前1,280℃（目标1,250℃）"
2. 点击查看详情，系统建议："减少燃气供给5%，增加空气供给3%"
3. 到窑炉现场，按建议手动调节阀门
4. 5分钟后温度回落至1,252℃，预警解除
5. 在小程序确认"已处理"，记录处理日志

## 4. 功能设计

### 4.1 系统架构

```
碳智优——数字化烧嘴管家
│
├── 硬件层
│   ├── 智能节能烧嘴（分段式喷火铜管 + 风叶优化）
│   ├── 传感器集群（温度/流量/压力/烟气成分，RS-485）
│   ├── PID气流调节模块（燃气/空气比例自动调节）
│   └── 工业网关（数据采集、边缘计算、4G/WiFi上传）

├── 边缘计算层

│
├── 云平台层
│   ├── 数据接入服务（MQTT/HTTP协议）

│   ├── 预警引擎（规则引擎 + 模型预测双驱动）
│   ├── 报告生成服务（日报/周报/月报）
│   └── 设备管理服务（OTA升级、状态监控）
│
├── 应用层
│   ├── 微信小程序（远程监测、预警接收、报告查看）
│   ├── PC端客户后台（多窑炉管理、数据导出、合同管理）
│   └── 管理后台（客户管理、设备管理、订单管理、模型管理）
│
└── 通知服务
    ├── 微信小程序推送（异常预警、报告推送）
    ├── 短信通知（紧急故障）
    └── 邮件通知（月度报告、合同文件）
```

#### 4.2.2 BiLG-BLS预测模块

**功能描述**：基于BiLG-BLS模型实时预测燃烧工况，为自动调控提供决策依据。

**模型架构**：

```
原始传感器数据
    ↓
VMD变分模态分解（分离高频火焰脉动 vs 低频热演变趋势）
    ↓
├─ LSTM流 ──→ 捕捉窑炉长周期热平衡惯性
│
└─ BiGRU流 ──→ 实时锁定风媒配比的瞬时物理耦合
    ↓
异构双流特征融合
    ↓
DCA自适应级联BLS（自动匹配最优映射层深度）
    ↓
预测结果（未来5-30分钟燃烧效率、温度趋势、异常概率）
    ↓
PID调控指令 / 异常预警
```

**业务规则**：

- 模型推理频率：每5分钟执行一次完整预测
- 预测时间窗口：未来5分钟（即时调控）、未来30分钟（趋势预警）
- 预测准确率目标：R² ≥ 0.919（即92%）
- 模型自动重训练：每周基于新数据增量更新
- 边缘端保留轻量级模型：网络中断时可本地推理基础预警

**模型效果对比**：

| 模型维度   | LSTM | GRU | 传统BLS | Transformer | **BiLG-BLS** |
| ------ | ---- | --- | ----- | ----------- | ------------ |
| 跨时序洞察  | ✓    | ✓   | ✕     | ✓           | ✓✓✓          |
| 多变量协同  | ✕    | ✕   | △     | ✓           | ✓✓✓          |
| 燃烧工况预测 | 71%  | 77% | 76%   | 85%         | **92%**      |
| 可解释性   | ✕    | ✕   | ✓✓    | △           | ✓✓✓          |
| 计算资源消耗 | 高    | 中   | 低     | 极高          | **低**        |

#### 4.2.3 异常预警模块

**功能描述**：基于规则引擎与BiLG-BLS预测结果，自动识别异常并推送预警。

**预警分级**：

| 级别    | 触发条件                   | 通知方式        | 响应要求    |
| ----- | ---------------------- | ----------- | ------- |
| 🔴 紧急 | 温度超限±50℃ / 燃气泄漏 / 设备离线 | 小程序推送+短信+电话 | 15分钟内响应 |
| 🟡 重要 | 效率下降>5% / 排放超标 / 传感器故障 | 小程序推送+短信    | 1小时内响应  |
| 🟢 提示 | 效率轻微波动 / 维护提醒 / 报告生成   | 小程序推送       | 24小时内查看 |

**预警处理流程**：

```
系统检测到异常
    ↓
判断预警级别
    ↓
├─ 紧急 ──→ [自动PID调控] + [推送告警]
│   ↓
│   人工确认处理 / 系统记录
│
├─ 重要 ──→ [推送预警] + [提供调控建议]
│   ↓
│   用户确认"已处理"或"忽略"
│
└─ 提示 ──→ [推送消息]，用户按需查看
```

**业务规则**：

- 同一异常30分钟内不重复推送
- 紧急预警未处理时，每15分钟追加提醒一次
- 预警处理记录存档，用于后续分析和模型优化
- 支持预警规则自定义（客户可调整阈值）

#### 4.2.4 自动调控模块

**功能描述**：接收BiLG-BLS预测结果，通过PID模块自动优化燃气/空气比例。

**调控逻辑**：

```
BiLG-BLS预测结果
    ↓
目标燃烧效率 / 目标温度
    ↓
PID控制器计算偏差
    ↓
输出调节指令（燃气阀开度、空气阀开度）
    ↓
下发至烧嘴执行机构
    ↓
传感器反馈实际值
    ↓
闭环调节（循环周期 ≤ 10秒）
```

**业务规则**：

- 自动调控默认开启，客户可手动切换为"建议模式"（只给建议，不自动执行）
- 单次调节幅度限制：燃气±3%，空气±5%，防止剧烈波动
- 调控后5分钟内持续监测，若效果不达预期则升级预警
- 记录每次调控的时间、原因、参数变化、效果评估

### 8.2 系统端异常

| 异常场景           | 处理策略               | 监控告警      |
| -------------- | ------------------ | --------- |
| 云平台服务宕机        | 边缘计算层接管基础调控，本地缓存数据 | 立即告警，启动灾备 |
| 数据库连接失败        | 启用只读副本，写入操作进入队列    | 立即告警      |
| BiLG-BLS模型推理超时 | 回退至规则引擎进行基础预警      | 模型服务告警    |
| 工业网关批量离线       | 排查网络/电力故障，通知客户     | 批量离线告警    |
| 高并发导致响应慢       | 启用CDN缓存，限流保护       | 响应时间>5秒告警 |
| 数据上传丢包         | 边缘端重传机制，记录丢包率      | 丢包率>1%告警  |
| 固件OTA升级失败      | 自动回滚至上一版本          | 升级失败告警    |

### 8.3 业务端异常

| 异常场景       | 触发条件         | 处理策略                                            |
| ---------- | ------------ | ----------------------------------------------- |
| 客户窑炉大修停产   | 连续7天无数据      | 暂停计费，标记"停产维护"，恢复后重新校准基线                         |
| 客户更换燃料类型   | 燃气成分变化       | 通知技术工程师重新标定模型参数                                 |
| 节能效果不达预期   | 连续30天节约率<5%  | 技术团队现场诊断，免费优化调整                                 |
| 客户拖欠服务费    | 逾期30天        | 暂停数据服务（保留监测），发送催缴通知 |
| 设备人为损坏     | 客户报告或传感器异常判断 | 判定责任，保修期内免费维修/更换，超出保修期报价维修                      |
| 基线数据异常     | 部署前数据不完整或波动大 | 延长基线采集期至14天，或采用行业基准值                            |
| 模型在新窑炉上效果差 | R²<0.85      | 启动针对性迁移学习，增加本地训练数据                              |

### 8.4 边界情况

- **新客户首次部署**：提供7天免费基线监测，建立能耗基准模型后再正式计费
- **多窑炉企业**：支持统一管理，各窑炉数据隔离，支持跨窑炉对比分析
- **技术工程师同时处理多客户告警**：按紧急程度排队，紧急告警优先响应
- **多用户同时操作同一设备**：乐观锁机制，后提交者提示"设备状态已更新"
- **合同到期前30天**：自动推送续约提醒，提供续约优惠
- **客户主动暂停服务**：保留数据6个月，恢复服务后数据无缝衔接
